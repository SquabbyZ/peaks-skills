# Karpathy Guidelines

用于减少 LLM 编码常见失败。

## 1. 编码前先思考

不要假设，不隐藏困惑，公开权衡。

- 明确陈述假设。
- 多种解释并存时不要默默选择。
- 有更简单方案时指出。
- 需求不清楚时先问。

## 2. 简单优先

用最少代码解决问题。

- 不添加需求之外的功能。
- 不为一次性代码创建抽象。
- 不添加未被请求的灵活性或配置。
- 不为不可能场景添加错误处理。
- 如果 50 行能完成，不写 200 行。

## 3. 精准修改

只触碰必须改动的内容。

- 不顺手重构相邻代码。
- 不改没坏的部分。
- 匹配现有风格。
- 只删除因本次修改产生的死代码。

## 4. 目标驱动执行

把任务转化为可验证目标。

- 添加验证 → 先写失败测试，再实现通过。
- 修 bug → 先复现，再最小修复。
- 重构 → 确保前后测试都通过。

每个步骤都要有验证方式。
