# /peaksbug - Bug 修复

输入 bug 的自然语言描述，自动完成从复现到修复的完整流程。

## 输入

{{input}}

示例：
- "登录按钮点击没反应"
- "用户列表加载很慢，要 5 秒"
- "支付接口返回 500 错误"

## 执行流程

### Phase 1: 信息收集（Grill）

**Step 1.1** — 复现问题
- 读取用户描述的操作步骤
- 尝试复现 bug
- 记录复现结果：成功 / 失败 / 信息不足

**Step 1.2** — 收集上下文
- 阅读相关代码（使用 MCP 或 Read/Grep）
- 查看日志、错误信息
- 了解相关模块的历史（git log）

**Step 1.3** — 分析归类

| 类别 | 说明 |
|------|------|
| 明确 bug | 能稳定复现，有明确错误信息 |
| 性能问题 | 响应慢、资源占用高 |
| 边界条件 | 空值、异常输入处理不当 |
| 信息不足 | 需要更多细节才能定位 |

```
┌─ Phase 1 完成：问题分析 ────────────────────────────┐
│                                                    │
│  问题类型：[类别]                                  │
│  复现状态：[成功/失败/需更多信息]                  │
│  相关模块：[文件/函数列表]                         │
│                                                    │
│  ✅ 进入 Phase 2 根因分析                          │
│  ❌ 信息不足 → 请求补充                            │
└────────────────────────────────────────────────────┘
```

---

### Phase 2: 根因分析（Diagnose）

**Step 2.1** — 追踪代码路径
- 从用户操作入口开始
- 追踪数据流和函数调用链
- 定位到具体的问题代码

**Step 2.2** — 确定根因
- 明确"为什么坏了"
- 区分直接原因和深层原因

**Step 2.3** — 验证假设
- 如果修复了会怎样？
- 会不会引入新问题？

```
┌─ Phase 2 完成：根因确定 ────────────────────────────┐
│                                                    │
│  根因：[具体原因描述]                              │
│  位置：[文件:行号 或 函数名]                       │
│  修复方案：[简要描述]                              │
│                                                    │
│  ✅ 进入 Phase 3 修复                              │
└────────────────────────────────────────────────────┘
```

---

### Phase 3: 修复实现（Fix）

**Step 3.1** — 编写修复代码
- 遵循 Karpathy Guidelines
- 只改必须改的
- 修复后清理自己造成的孤立代码

**Step 3.2** — 编写/更新测试
- 编写能复现 bug 的测试（RED）
- 修复代码让测试通过（GREEN）
- 确保测试覆盖边界条件

**Step 3.3** — 执行测试验证
```bash
# 运行相关测试
npm test -- --grep "相关测试名"

# 运行 E2E（如有）
npx playwright test
```

---

### Phase 4: Code Review + 安全检查

**Step 4.1** — Code Review
```
┌─ Code Review ─────────────────────────────────────┐
│  调用 code-reviewer agent                         │
│  重点：修复是否正确、是否有副作用                  │
│                                                    │
│  ✅ 通过 → 进入安全检查                            │
│  ❌ 失败 → 修复后重新 CR                          │
└────────────────────────────────────────────────────┘
```

**Step 4.2** — 安全检查
```
┌─ 安全检查 ─────────────────────────────────────────┐
│  调用 security-reviewer agent                      │
│  重点：修复是否引入安全漏洞                        │
│                                                    │
│  ✅ 通过 → 进入 Phase 5 验证                       │
│  ❌ 失败 → 修复                                   │
└────────────────────────────────────────────────────┘
```

---

### Phase 5: 验证 + 回归测试

**Step 5.1** — 验证修复
- 重新执行用户报告的操作步骤
- 确认 bug 已修复

**Step 5.2** — 回归测试
- 运行相关模块的测试套件
- 确保修复没有破坏其他功能

**Step 5.3** — 更新文档
- 如需更新 README、API 文档

---

### Phase 6: 修复报告

输出修复报告到 `.peaks/bugs/bug-[日期].md`：

```
# Bug 修复报告

## 问题描述
[用户报告的现象]

## 根因分析
- 根因：[具体原因]
- 位置：[文件:行号]

## 修复方案
- 修复内容：[做了什么修改]
- 验证方式：[如何确认修复]

## 测试结果
- [ ] 复现测试通过
- [ ] 回归测试通过
- [ ] Code Review 通过
- [ ] 安全检查通过

## 相关文件
- [修改的文件列表]

修复完成时间：[日期]
```

---

## 快速模式

如果 bug 很简单（如明显的空指针、拼写错误）：

```
/peaksbug 按钮文字写错了，应该是"提交"不是"提交1"
```

直接：
1. 定位并修复
2. 简单测试验证
3. 提交

```
✅ Bug 修复完成

产出物：
  - .peaks/bugs/bug-[日期].md
  - 修复的代码（已提交）
  - 测试验证通过
```

---

## 提示

- 如果 bug 无法复现，会提示用户协助提供更多信息
- 涉及支付、数据安全等敏感操作时会特别谨慎
- 修复完成后建议用户进行验收确认
