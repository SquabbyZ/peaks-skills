# 把一个大 Skill 拆成一支 AI 工程小队

如果一个开发者只记得这几个命令：

```text
peaks-solo
peaks-prd
peaks-ui
peaks-rd
peaks-qa
peaks-sc
peaks-txt
```

他应该就能把一个真实项目从“我想重构一下”推进到“目标清楚、方案可比、回归有保护、变更有证据、上下文可继承”。

这就是 Peaks 短名技能家族想解决的问题。

它不是把 `peaks-sdd` 继续做大，也不是再发明一个更复杂的流程图。它更像是把一个越来越重的 AI 开发助手，拆成一组能独立工作的角色技能：产品、UI、研发、QA、同步、上下文，再用一个全自动入口把它们串起来。

## 旧问题：一个 Skill 很容易长成一坨流程

`peaks-sdd` 最初的价值很清楚：用户用自然语言描述需求，系统按 Spec-Driven Development 的方式推动项目。

它可以做初始化、PRD、设计、技术方案、agent 调度、代码实现、测试、安全审查、QA 报告。对于小型端到端任务，这种单入口体验很好。

但问题也出在这里。

当一个 Skill 同时负责产品、设计、研发、QA、上下文、同步和发布，它很快会变成一个“全能单体”：

- 产品产物和技术产物混在一起；
- harness 门禁只属于总流程，不属于具体角色；
- OpenSpec 只像是开发阶段工具，而不是每个能力自己的规格系统；
- 子 agent 做了很多事，但角色边界不稳定；
- 后续要升级某一段能力时，很容易牵动整个 Skill；
- 用户也很难说清楚自己要用的是产品能力、重构能力、测试能力，还是全自动编排能力。

所以新方向不是“加强 `peaks-sdd`”，而是把它旁边生长出一组新的短名技能。

旧的 `skills/peaks-sdd/` 保留，作为历史实现、harness 经验和迁移参考。新的体系在 `skills/` 下新建目录，不原地拆旧目录。

目标结构是：

```text
skills/
├── peaks-sdd/       # 保留，不原地重构
├── peaks-solo/      # 新全自动编排入口
├── peaks-prd/       # 产品与需求技能
├── peaks-ui/        # UI/UX 与体验技能
├── peaks-rd/        # 研发与重构技能
├── peaks-qa/        # 测试与验收技能
├── peaks-sc/        # 同步、审批、变更控制技能
└── peaks-txt/       # 上下文胶囊与知识压缩技能
```

## 新思路：每个子技能都是一等产品

Peaks 短名技能家族的核心原则是：每个子技能都能独立使用，也能被编排。

这句话很重要。

`peaks-prd` 不应该只是 `peaks-solo` 里面的一段 prompt。它自己就应该能完成产品澄清、PRD、验收标准和需求变更。

`peaks-qa` 也不应该只是最后跑一下测试。它自己就应该能读取需求、设计和实现证据，生成测试矩阵、回归范围、自动化补丁和验收报告。

每个子技能至少应该有自己的：

- `SKILL.md`：入口、触发语、工具边界、执行原则；
- `references/`：方法论、产物模板、命令迁移说明；
- `templates/`：agent、harness、OpenSpec 模板；
- `schemas/`：artifact、evidence、trace、context schema；
- `scripts/`：校验、格式化、迁移、质量门禁脚本；
- `examples/` 或 `fixtures/`：dogfood 和回归样例。

这让每个 Skill 都可以独立升级。

如果只想写一个 PRD，用 `peaks-prd`。如果只想给现有项目做回归保护，用 `peaks-qa`。如果只想压缩项目上下文，用 `peaks-txt`。如果想让 AI 自动跑完整重构流程，再用 `peaks-solo`。

## 七个短名分别做什么

### `peaks-solo`：全自动单人编排入口

`peaks-solo` 是新的自动入口，但它不是旧 `peaks-sdd` 的原地改名。

它负责识别场景、推荐模式、调用子技能、追踪产物、汇总报告。

它不应该直接替代产品、UI、研发、QA、同步和上下文技能。它更像一个调度员，而不是一个把所有工作都塞进自己身体里的超级 agent。

它可以识别这些场景：

- 重构；
- 功能开发；
- bug 修复；
- QA 强化；
- 发布验证；
- 事故响应。

但第一阶段，它优先跑通一个真实场景：重构。

### `peaks-prd`：把意图变成可验收目标

`peaks-prd` 负责产品意图、PRD、验收标准和变更说明。

在新产品开发里，它可以写完整 PRD。

在重构场景里，它不需要写一份臃肿的新产品文档。它只需要回答几个关键问题：

- 为什么要重构；
- 哪些行为必须保持不变；
- 哪些问题必须改善；
- 哪些范围明确不做；
- 用什么证据证明重构成功。

这会变成重构专用 PRD。它不是形式主义文档，而是后面 `peaks-rd` 和 `peaks-qa` 的输入。

### `peaks-ui`：体验、交互和原型

`peaks-ui` 负责 UI/UX、交互、视觉方向、高保真 HTML 原型和体验评估。

它不是每次都必须介入。

如果一次重构只涉及后端模块、数据结构或测试体系，`peaks-ui` 可以不参与。如果重构涉及页面结构、设计系统、表单体验、交互流或视觉一致性，它就应该进入流程。

它的输出不是“好看的图”，而是能被研发和 QA 消费的体验产物：页面状态、交互流、组件边界、文案、可访问性要求和 UI 回归种子。

### `peaks-rd`：技术分析、方案比较和实现

`peaks-rd` 是 Refactor-first MVP 里的主执行技能。

它要先读现状，而不是上来就改代码。

一个合格的 `peaks-rd` 重构流程应该先输出：

- 当前状态；
- 技术债和边界；
- 2-3 个重构候选方案；
- 每个方案的成本、收益、风险和回滚路径；
- 受影响模块；
- 初步任务图。

然后等用户选择方案。

再然后等 `peaks-qa` 建立回归保护。

最后才进入实现。

这条顺序能避免 AI 最常见的问题：在没有验收标准、没有回归保护、没有用户确认的情况下，把一个项目“热心地重构坏”。

### `peaks-qa`：先保护，再验证

`peaks-qa` 不只是“最后测试一下”。

在重构场景里，它必须先于实现介入。

它要从 `peaks-prd` 读取重构目标，从 `peaks-rd` 读取风险模块，然后生成：

- 回归矩阵；
- 基线报告；
- 必须补的测试；
- 手工检查清单；
- 无法自动验证的风险点。

重构完成后，再用同一套矩阵回归验证。

这让重构变成一个可证明的过程，而不是“我感觉代码更干净了”。

### `peaks-sc`：同步、审批和变更影响

`peaks-sc` 的名字来自 source control / sync / change control。

它负责记录变更如何穿过产物和代码：

- 哪些模块受影响；
- 哪些文件被改动；
- 哪些产物版本参与了这次重构；
- 用户选择了哪个方案；
- 哪些风险被接受；
- 哪些测试证据证明行为没有破坏；
- 代码仓和产物仓指针是什么。

它也可以吸收现有 commit、PR、checkpoint、Jira 相关 command 的能力。

但它不是产品技能，不写 PRD；不是研发技能，不实现代码；不是 QA 技能，不执行测试。

它的价值是让一次 AI 参与的变更有可追踪的证据链。

### `peaks-txt`：上下文胶囊和长期知识

AI 工程协作最容易浪费的是上下文。

一个 agent 做完产品脑暴，另一个 agent 做技术设计，再一个 agent 做测试。如果每次都把完整对话塞给下游，既浪费 token，也容易污染判断。

`peaks-txt` 负责把上下文压成可以传递的胶囊。

它记录：

- 用户原始目标；
- 已确认决策；
- 假设；
- 已放弃方案；
- 风险；
- 角色切片；
- 过期条件。

研发看到的是研发需要的上下文。QA 看到的是 QA 需要的上下文。UI 看到的是体验相关上下文。

这比“把整段聊天记录给下一个模型”更接近团队协作。

## 为什么第一场景是重构

第一阶段不从 0→1 新项目开始，而是从真实项目重构开始。

不是因为 0→1 不重要，也不是因为完整架构做不了。

相反，重构是更适合验证 Skill family 边界的场景。

一个真实项目重构会同时考验这些能力：

- AI 能不能读懂现有代码；
- `peaks-prd` 能不能把“想重构”变成明确目标；
- `peaks-rd` 能不能提出多个方案，而不是直接开改；
- `peaks-qa` 能不能先建立回归保护；
- `peaks-sc` 能不能记录影响范围和证据；
- `peaks-txt` 能不能把过程沉淀成可复用上下文；
- `peaks-solo` 能不能只做编排，而不重新长成单体。

这条链路跑通后，再扩展到 0→1、bugfix、QA 强化、发布验证和事故响应，会更稳。

Refactor Mode 的理想流程是：

```text
peaks-solo refactor
  → peaks-prd 定义重构目标
  → peaks-txt 生成项目上下文胶囊
  → peaks-rd 分析项目并提出 2-3 个重构方案
  → peaks-qa 生成回归保护
  → 用户选择方案
  → peaks-rd 执行最小安全重构
  → peaks-qa 回归验证
  → peaks-sc 记录变更影响与同步状态
  → peaks-txt 归档上下文胶囊
  → peaks-solo 汇总最终报告
```

这不是为了流程完整而流程完整。

它的目标很直接：让 AI 重构从“大胆改代码”变成“有目标、有方案、有保护、有证据”。

## Harness 和 OpenSpec 不再只属于总入口

旧的 `peaks-sdd` 已经沉淀了很多有价值的门禁：PRD 确认、设计确认、技术方案确认、代码审查、安全审查、QA 报告、最终报告。

新设计不会丢掉这些经验。

但它们不应该只存在于 `peaks-solo` 里。

每个子技能都应该有自己的 harness。

`peaks-prd` 有需求完整度和验收标准门禁。`peaks-ui` 有原型、交互和设计确认门禁。`peaks-rd` 有方案比较、任务图、实现和审查门禁。`peaks-qa` 有回归矩阵、覆盖率、冒烟和验收门禁。`peaks-sc` 有同步、审批和冲突门禁。`peaks-txt` 有上下文一致性和过期检查。

OpenSpec 也一样。

OpenSpec 不应该只是“功能开发阶段用一下”。每个子技能都可以用 OpenSpec 管理自己的能力变化：

- `peaks-prd` 管 PRD 模板和验收标准协议；
- `peaks-ui` 管原型产物和体验评估规则；
- `peaks-rd` 管重构模式、任务图和实现门禁；
- `peaks-qa` 管测试矩阵和验收报告；
- `peaks-sc` 管同步、审批和 change-impact；
- `peaks-txt` 管 context capsule 和知识沉淀；
- `peaks-solo` 管场景路由和跨技能 handoff。

这样做的好处是：每个技能的升级都有自己的规格和验收，而不是所有变化都挤进一个总入口。

## 为什么要保持模型中立

这个体系会用 Claude 的现有 commands、agents 和 skills 作为迁移素材。

但 Peaks 的产物协议不应该绑定 Claude。

原因很简单：这套技能家族本身计划按全 AI 工程模式开发，并且会使用 MiniMax 2.7 作为主要实现模型之一。

如果协议只适配 Claude 的上下文、工具调用或命令习惯，后面就会变成模型锁定。

所以新的产物应该尽量模型中立：

- `manifest` 描述产物身份、版本、状态和输入输出；
- `context capsule` 描述角色可消费的最小上下文；
- `approval record` 描述用户确认和风险接受；
- `change impact` 描述模块、文件、测试和同步影响；
- `evidence` 保存截图、日志、测试结果和审查记录。

Claude 可以读，MiniMax 2.7 也应该能读。其他模型或工具以后也可以读。

这才是“AI 协作协议”，而不是某个模型的私有工作流。

## 全 AI 开发，不等于不要架构

这套技能家族会用全 AI 模式开发。

这会改变 MVP 的定义。

传统 MVP 经常意味着：为了节省人工开发周期，先砍掉架构，只做最小功能。

但在全 AI 开发模式下，真正稀缺的不是写文件的人力，而是验证、边界和反馈。

所以这里的 MVP 不是“少做一点”。

它是“先选择最能验证架构的闭环”。

也就是 Refactor-first。

可以同时设计 7 个子技能的骨架、协议、harness 和 OpenSpec；可以让多个 agent 并行生成、互审、修复；可以用真实项目 dogfood；可以用 eval 和质量门禁收敛。

但每一部分都要有可验证产物。

没有证据的复杂度只是噪音。有 dogfood 反馈的复杂度，才可能变成架构。

## 现有 Claude commands 怎么处理

新体系采用“重命名优先”。

也就是说，用户以后主要记 `peaks-*` 短名，而不是继续记一堆历史 command。

现有 command 的处理方式分三类。

第一类，直接迁移：

```text
prp-prd        → peaks-prd
product        → peaks-prd
design         → peaks-ui
prp-implement  → peaks-rd
build-fix      → peaks-rd
testing        → peaks-qa
quality-gate   → peaks-qa
prp-commit     → peaks-sc
save-session   → peaks-txt
multi-workflow → peaks-solo
```

第二类，拆分迁移。

比如 `feature-dev` 这种 command，本身跨了产品、研发、测试和编排。它不应该整体搬到某一个新技能里，而应该拆开：场景识别归 `peaks-solo`，需求归 `peaks-prd`，实现归 `peaks-rd`，验证归 `peaks-qa`。

第三类，不进入核心家族。

一些 legacy shim、底层运维、语言专用 build/review/test command，可以作为 adapter 或平台能力存在，但不应该成为用户必须记住的 Peaks 主入口。

这会让用户心智保持简单：

```text
solo / prd / ui / rd / qa / sc / txt
```

七个词够了。

## 这套东西最终想交付什么

第一阶段的交付不是一个宏大的平台。

它应该是这些具体东西：

1. 新建 7 个 `skills/peaks-*` 子技能目录；
2. 每个子技能有清晰的 `SKILL.md`；
3. 每个子技能能独立运行，也能被 `peaks-solo` 编排；
4. 每个子技能有自己的 harness 和 OpenSpec 作用点；
5. `peaks-solo refactor` 有完整流程；
6. `manifest`、`context capsule`、`approval record`、`change impact` 有最小协议；
7. 现有 Claude commands 有迁移表；
8. 在一个真实项目上跑完一轮重构 dogfood。

跑完以后，再决定哪些边界要调整。

如果 `peaks-rd` 太重，就拆 adapter。如果 `peaks-qa` 介入太晚，就把回归保护提前。如果 `peaks-txt` 生成的上下文太泛，就收紧角色切片。如果 `peaks-solo` 又开始吞职责，就把逻辑退回子技能。

这才是技能家族的正确进化方式：不是一次设计完美，而是每个子技能都能独立变好。

## 结语：AI 工程协作需要更小的角色边界

很多 AI coding workflow 的问题，不是模型不够强，而是边界太粗。

一个大 prompt 负责所有事，一个大 agent 扛完整流程，一个大 skill 既写 PRD 又写代码又跑测试。短期看起来快，长期很难维护。

Peaks 短名技能家族的判断是：AI 工程协作应该往更小、更清晰、可独立升级的角色边界走。

`peaks-solo` 负责串起来。

`peaks-prd` 负责说清楚要什么。

`peaks-ui` 负责体验怎么成立。

`peaks-rd` 负责技术上怎么安全改变。

`peaks-qa` 负责证明没有改坏。

`peaks-sc` 负责记录变更证据和同步关系。

`peaks-txt` 负责把上下文变成可传递资产。

第一步就从重构开始。

因为真实项目里的重构最诚实：目标不清楚会暴露，测试不足会暴露，边界混乱会暴露，AI 乱改代码也会暴露。

如果这套技能家族能把重构做成一个有目标、有方案、有保护、有证据的闭环，它就有资格继续扩展到更大的 AI 工程协作。
