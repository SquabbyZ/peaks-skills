# Peaks AI 协作技能体系设计

日期：2026-05-14
状态：评审草案

## 1. 目的

当前的 `peaks-sdd` 已经能够完成小型端到端业务请求，但团队协作会暴露更大的问题：角色边界不清晰、交接产物没有协议化、上下文不可移植，并且产品、设计、研发、QA 与事故响应之间的下游变更很难同步。

本设计已经不再只是 `peaks-sdd`、harness 或 OpenSpec 的增强方案，而是一个围绕角色技能、共享产物、上下文胶囊、同步、问题处理和持续技能沉淀构建的、厂商中立的 Peaks AI 协作技能体系。`peaks-sdd` 应该成为该体系中的编排入口之一，而不是承载所有职责的单体技能。

目标是真实生产力，而不是为了流程完整而追求流程完整：

- 低学习成本：用户用自然语言描述工作。
- AI 推荐工作流模式和技能组合，但由用户决策。
- 中间产物降低沟通成本。
- 群体执行降低交付与探索时间。
- 面向人阅读和面向模型读取的产物保持同步。
- 输出保持模型厂商中立，并可跨工具复用。
- 技能增强人，而不是取代专业判断。

## 2. 产品哲学

### 2.1 人在回路中的增强

Peaks 不是为了替代产品经理、设计师、工程师、QA 或发布负责人而设计的。

技能可以自动化：

- 起草；
- 分析；
- 编码；
- 测试；
- 同步；
- 证据收集；
- 报告生成；
- 候选方案探索；
- 影响分析；
- 归档。

人类保留以下事项的最终权力：

- 产品决策；
- 架构决策；
- UI/UX 验收；
- QA 验收；
- 发布风险接受；
- 生产事故严重等级；
- 回滚/降级批准。

报告必须区分：

- `AI Verified`：已通过自动化方式验证，并附带证据；
- `Human Required`：必须由人审查；
- `Human Confirmed`：已由人明确确认；
- `Risk Accepted`：已知风险已由人接受。

### 2.2 抬高下限原则

Peaks 应该提高协作的最低质量，而不是保证每个用户都能获得相同的最高水平结果。不同用户、行业、工具、模型、MCP 可用性和使用习惯都会产生不同结果。

系统应该：

- 帮助经验不足的用户避免遗漏关键步骤；
- 帮助经验丰富的用户减少重复劳动；
- 减少跨角色协作中可避免的沟通损耗；
- 让 AI 输出更贴近用户输入和业务上下文。

### 2.3 适配上下文原则

技术、产品设计和测试方法本身没有绝对的好坏。它们只是更适合或不适合当前业务、约束、团队成熟度、交付场景、风险容忍度和维护能力。

每个选项都应该从以下维度评估：

- 业务适配度；
- 用户体验；
- 实现成本；
- 维护成本；
- 团队熟悉度；
- 成熟度；
- 回滚路径；
- QA 覆盖可行性；
- 交付紧急程度；
- 长期价值。

### 2.4 边界拓展原则

技能应该帮助用户突破当前经验边界，而不是替代用户决策。

对于产品、设计、工程、QA 和重构，角色技能应该呈现：

1. 用户直觉上或熟悉的做法；
2. 成熟的行业替代方案；
3. 质量更高但成本也更高的方案；
4. 标记为 `use with caution` 的前瞻性方案；
5. 不推荐的方案及原因；
6. 需要人类决策的节点。

前沿方法必须标注成熟度、风险、前置条件以及不适用场景。

## 3. 技能产品家族

### 3.1 角色技能

系统应该拆分为面向角色的技能。每个技能既支持独立使用，也支持编排使用。

| 技能 | 主要职责 |
| --- | --- |
| `peaks-product` | 产品脑暴、PRD、验收标准、需求变更、产品上下文胶囊 |
| `peaks-design` | UI/UX、交互、视觉方向、高保真 HTML 原型、用户体验评估与体验产物 |
| `peaks-rd` | 架构选项、技术设计、任务拆解、实现、单元测试、审查、安全 |
| `peaks-qa` | 测试矩阵、测试用例、自动化补丁、回归范围、验证报告 |
| `peaks-issue` / `peaks-incident` | QA 问题、生产事故、复现产物、缓解、修复、复盘 |
| `peaks-artifact` | 厂商中立的产物协议、Schema 校验、差异、追踪、影响分析 |
| `peaks-context` | 上下文胶囊生成、压缩、角色切片、过期检查 |
| `peaks-sync` | 独立产物 Git 仓与业务代码仓同步、提案、审批、冲突检测 |
| `peaks-sdd` | 兼容入口与编排器之一：场景识别、模式推荐、跨技能编排、最终匹配报告 |

### 3.2 独立模式与编排模式

每个角色技能必须支持两种模式。

独立模式示例：

- 产品经理使用 `peaks-product` 生成 PRD 包；
- 设计师使用 `peaks-design` 探索交互选项；
- 工程师使用 `peaks-rd` 设计并实现技术变更；
- QA 使用 `peaks-qa` 创建测试矩阵和报告。

编排模式示例：

- `peaks-sdd` 为新项目运行产品 → 设计 → 研发 → QA；
- `peaks-sdd` 将 QA 问题路由给研发修复，再交回 QA 回归；
- `peaks-sdd` 运行事故模式，并协调研发、QA 和复盘输出。

### 3.3 每个角色技能内部的群体执行

每个角色技能在有价值时都应该能够使用群体执行。

示例：

- `peaks-product`：需求澄清者、用户故事分析者、验收标准审查者、领域风险分析者；
- `peaks-design`：UX 流程设计师、视觉方向代理、可访问性审查者、原型生成器；
- `peaks-rd`：架构师、前端、后端、数据库、代码审查者、安全审查者；
- `peaks-qa`：风险分析者、测试矩阵生成器、自动化代理、回归分析者、性能审查者；
- `peaks-issue`：复现代理、日志分析者、影响分析者、缓解规划者、复盘撰写者。

## 4. 工作流模式

`peaks-sdd` 应该推断场景并推荐模式。最终决策属于用户。

### 4.1 场景

支持的目标场景：

- `0-to-1`：新产品/新项目；
- `1-to-N`：现有项目中的功能迭代；
- bug 修复；
- 项目重构；
- QA 强化；
- 发布验证；
- QA 问题处理；
- 生产事故响应；
- 复盘与学习；
- 技能沉淀。

### 4.2 执行模式

| 模式 | 适用场景 | 提速策略 |
| --- | --- | --- |
| Fast Mode | 小型或紧急的低风险工作 | 精简流程 |
| Standard Mode | 常规功能工作 | 保留关键产物和门禁 |
| Team Mode | 多角色协作 | 完整角色交接 |
| Strict Mode | 高风险/合规/审计工作 | 强门禁和证据 |
| Parallel Candidate Mode | 用户无法在多个选项中选择 | 并行生成候选方案/原型 |
| Incident Mode | QA 阻塞问题或生产事故 | 降低影响、诊断、修复、验证、复盘 |

重要规则：

- 小请求通过精简流程变快。
- 大型紧急请求通过群体并行变快，而不是盲目削减质量。

### 4.3 单人工作流兼容性

Peaks 支持将团队工作流折叠为单人模式。同一套角色技能、产物、上下文胶囊、追踪链接和验证门禁会被复用。差异在于审批拓扑，而不是产物语义。

在团队模式中：

- 产品确认 PRD；
- 设计确认原型和 UX 方向；
- 研发确认架构和实现风险；
- QA 确认验证和发布证据。

在单人模式中，一个用户以对应角色的代理负责人身份，依次经过相同的角色视角并确认每个检查点。

单人模式变体：

| 模式 | 适用场景 | 行为 |
| --- | --- | --- |
| Solo Fast | 小型低风险请求 | 最少角色检查和轻量产物 |
| Solo Standard | 常规单人功能工作 | 保留产品/设计/研发/QA 视角，但确认更轻量 |
| Solo Strict | 单人但高风险工作 | 即使同一个人确认，也保留强门禁 |
| Solo Parallel Candidate | 单人用户缺乏足够经验做选择 | 并行生成候选选项/原型用于比较 |

单人模式应该保留可升级性：一个由单人启动的项目之后可以迁移到团队模式，因为产物、上下文胶囊和追踪链接遵循同一协议。

## 5. Peaks 协作协议

Peaks 需要一个类似 MCP 的团队产物协议。该协议必须保持模型厂商中立，且不能依赖 Claude/Anthropic 特定格式。

### 5.1 产物结构

每个主要产物都应该有两个同步表示：

```text
artifact.<id>/
├── human.md
├── model.json
├── manifest.json
├── schema.json
├── trace.json
├── context/
└── evidence/
```

- `human.md`：供人审查，适合放入独立产物 Git 仓，支持自然语言修改。
- `model.json`：供技能、代理、CI 和其他模型使用的结构化表示。
- `manifest.json`：id、version、status、owner、source、hash、Git 指针。
- `schema.json`：用于校验的 JSON Schema。
- `trace.json`：需求、决策、设计、代码、测试、问题与发布之间的关系。
- `context/`：上下文胶囊和角色专用切片。
- `evidence/`：截图、录屏、日志、测试结果、审查记录。

### 5.2 双层 Schema

使用双层协议：

- 用于互操作的固定核心字段；
- 用于领域特定需求的灵活扩展字段。

核心字段示例：

```json
{
  "protocolVersion": "peaks-collaboration/v1",
  "artifactId": "prd.member-management.v1",
  "artifactType": "prd",
  "ownerSkill": "peaks-product",
  "scenario": "1-to-N-feature",
  "status": "draft",
  "summary": "...",
  "intent": {
    "userInput": "...",
    "goals": [],
    "nonGoals": [],
    "constraints": []
  },
  "contracts": {
    "consumableBy": ["peaks-rd", "peaks-qa", "peaks-sdd"],
    "requires": [],
    "provides": []
  },
  "traceability": {
    "sourceInputs": [],
    "derivedArtifacts": [],
    "decisions": [],
    "openQuestions": []
  },
  "acceptance": {
    "criteria": [],
    "risks": []
  },
  "extensions": {}
}
```

### 5.3 人类/模型双向同步

当用户要求修改面向人阅读的产物时，系统必须：

1. 修改独立产物 Git 仓中的 `human.md`；
2. 实时且一致地更新 `model.json`；
3. 运行 Schema 校验；
4. 运行人类/模型一致性检查；
5. 计算下游影响；
6. 生成差异摘要；
7. 当产物已批准或面向评审时，请求人类确认。

## 6. 上下文胶囊协议

仅有产物还不够。团队还需要可移植且精确的上下文。

### 6.1 为什么需要上下文胶囊

下游角色可能需要调用上游技能。示例：研发使用 `peaks-product` 修订 PRD 中不清晰的部分。研发没有原始产品 VibeCoding 上下文，但倾倒完整产品对话既嘈杂也不安全。

上下文胶囊保留最小但有用的背景。

### 6.2 生成模型

同时使用：

- 角色工作流中的实时捕获；
- 阶段结束后的压缩和校准。

### 6.3 胶囊内容

上下文胶囊应该包含：

- 原始请求；
- 业务目标；
- 成功定义；
- 已确认决策；
- 假设；
- 已放弃选项；
- 未解决歧义；
- 角色专用视图；
- 过期与失效条件。

示例：

```json
{
  "capsuleVersion": "peaks-context/v1",
  "sourceSkill": "peaks-product",
  "artifactId": "prd.member-management.v1",
  "audience": ["peaks-rd", "peaks-qa", "peaks-product"],
  "purpose": "Enable downstream roles to understand and safely revise the PRD.",
  "confirmedDecisions": [],
  "assumptions": [],
  "discardedOptions": [],
  "ambiguities": [],
  "roleViews": {
    "rd": { "mustKnow": [], "niceToKnow": [] },
    "qa": { "mustKnow": [], "riskFocus": [] }
  },
  "staleness": { "lastUpdatedAt": "", "invalidatedBy": [] }
}
```

### 6.4 上下文切片

上下文必须按任务和角色切片。

示例：

- 研发修订产品逻辑时，需要目标、决策、假设、已放弃选项和技术约束。
- QA 生成测试用例时，需要验收标准、状态、风险、交互流程和边界情况。
- 设计审查 C 端页面时，需要产品目标、用户情绪、品牌方向和目标行为。

不要在角色之间盲目传递完整对话。

## 7. 独立产物 Git 仓与业务代码仓同步

### 7.1 事实源模型

新的协作方案应该使用独立产物 Git 仓承载中间产物、协作记录、上下文胶囊、确认记录和证据；业务代码仓只承载工程实现与可执行资产。这样可以让产品/设计/QA/事故产物脱离具体业务代码仓复用，也可以通过 Git 的分支、提交、差异和审查机制管理人类/模型同步。

独立产物 Git 仓主要用于：

- 产品脑暴；
- PRD；
- 设计说明；
- 高保真原型评审；
- 需求评审记录；
- 产品变更提案；
- 人类确认记录；
- 上下文胶囊；
- 变更影响包；
- QA 测试计划、矩阵、验收报告；
- 问题/事故复现证据与复盘；
- 从 `.peaks/changes/<change-id>/` 迁移而来的中间产物。

业务代码仓主要用于：

- 工程实现代码；
- 单元测试；
- 自动化测试；
- 可执行脚本；
- CI 配置；
- 与代码强绑定的审查/安全/覆盖率/运行时证据。

QA 产物采用双仓协作：

- 独立产物 Git 仓保存测试计划、矩阵、验收报告和人工确认记录；
- 业务代码仓保存自动化测试、覆盖率、运行时冒烟和可供 CI 消费的证据；
- 两者通过 trace、manifest 和提交哈希互相引用。

### 7.2 同步元数据

产物快照应该存储业务代码仓指针，而不是假装产物仓是唯一事实源。业务代码仓也应该能反向指向对应的产物版本。

`remote.json` 示例：

```json
{
  "provider": "git",
  "artifactRepo": "git@example.com:org/project-artifacts.git",
  "artifactBranch": "feature/member-management",
  "artifactCommit": "...",
  "codeRepo": "git@example.com:org/project-app.git",
  "codeBranch": "feature/member-management",
  "codeCommit": "...",
  "lastSyncedAt": "...",
  "contentHash": "..."
}
```

### 7.3 提案与审批流程

当研发发现产品内容不清晰时：

1. 研发在独立产物 Git 仓创建一个产品澄清问题或变更分支。
2. `peaks-sync` 读取产物仓中的 PRD、人类文档、模型产物、上下文胶囊，以及业务代码仓中的技术约束。
3. `peaks-product` 生成变更提案，并同时更新 `human.md` 与 `model.json`。
4. 产物仓通过 Git diff 展示差异供产品审查。
5. 产品确认或拒绝。
6. 已确认变更以产物仓提交的方式固化，并更新人类文档、模型产物、上下文胶囊和业务代码仓指针。
7. 变更影响传播到研发和 QA。

只要变更还没有提交或合并到产物仓的正式分支，就不会影响团队认可的事实源。第一阶段应该优先实现协议、Git 连接器和 CLI 能力；桌面端或本地 Web 同步控制台可以之后再增加，用于查看差异、审批、冲突和同步状态。

## 8. 体验产物协议

产品和设计产物应该是体验性的，而不只是文本性的。

### 8.1 结构

```text
experience/
├── prototype.html
├── layout.model.json
├── interaction-flows.json
├── page-states.json
├── component-map.json
├── copy-and-microcopy.md
├── qa-scenarios.seed.json
└── implementation-hints.md
```

### 8.2 按角色划分的价值

产品：

- 在使用原型时发现边界情况；
- 评估交互和文案；
- 判断体验是否匹配业务意图。

设计/UI：

- 评估视觉层级、吸引力、情绪质量、可访问性和交互质量。

前端：

- 消费布局和交互产物，以便更精确地实现。

后端：

- 理解状态、权限、API 期望、校验和流转。

QA：

- 从状态、交互和验收标准生成测试矩阵与用例；
- 基于更快的业务理解审查生成的测试用例。

### 8.3 B 端与 C 端设计权重

B 端产品通常优化：

- 可用；
- 高效；
- 清晰；
- 低错误率；
- 高信息密度。

C 端产品通常要求功能性、可用性、视觉吸引力、情绪质量和用户体验具有相近的重要性。

`peaks-sdd` 应该根据产品类型和风险决定运行轻量设计流程还是强设计流程。

## 9. 变更传播协议

产品变更必须传播到研发和 QA。

产品变更确认后，系统应该生成：

- 研发影响包；
- QA 影响包；
- 更新后的产物；
- 受影响的下游产物列表；
- 人工审查清单。

QA 影响应该包含：

- 测试矩阵更新；
- 需要修改的用例；
- 需要新增的用例；
- 需要废弃的用例；
- 受影响的自动化脚本；
- 受影响的验收检查；
- 面向测试人员的变更说明。

示例：

```json
{
  "artifactType": "change-impact",
  "changeId": "chg.member-expiry-policy.v2",
  "sourceArtifact": "prd.member-management.v1",
  "changedRequirements": [],
  "changedAcceptanceCriteria": [],
  "impactedDomains": ["membership", "permission", "notification"],
  "qaImpact": {
    "testMatrixUpdates": [],
    "casesToModify": [],
    "casesToAdd": [],
    "casesToRetire": [],
    "automationScriptsAffected": []
  }
}
```

## 10. 问题与事故协议

QA 问题和生产事故应该共享一个通用的问题产物协议。生产事故是同一问题处理链中严重性更高、紧急度更高的实例。

### 10.1 QA 问题产物

```text
issue.<id>/
├── human.md
├── model.json
├── reproduce.md
├── expected-actual.md
├── requirement-links.md
├── environment.json
├── evidence/
│   ├── screenshot.png
│   ├── recording.webm
│   ├── console.log
│   ├── network.har
│   └── server.log
└── regression.md
```

### 10.2 研发消费流程

`peaks-rd` 应该消费问题产物，以便：

1. 阅读复现步骤和证据；
2. 复现问题或回放证据；
3. 将问题链接到需求、设计、技术任务和代码；
4. 缩小可能的问题区域；
5. 实现最小修复；
6. 生成修复产物；
7. 交回 QA 做回归。

### 10.3 生产事故流程

事故模式优先级：

1. 降低影响；
2. 保护数据；
3. 诊断根因；
4. 缓解、降级、回滚或热修；
5. 验证恢复；
6. 生成复盘；
7. 更新产品/研发/QA 产物；
8. 沉淀可复用技能或检查。

事故输出应该包含：

- 事故摘要；
- 影响范围；
- 时间线；
- 怀疑根因与确认根因；
- 缓解计划；
- 修复计划；
- 验证证据；
- 复盘；
- 后续任务；
- 回归补充；
- 技能候选项。

## 11. 并行候选模式

当用户缺乏足够经验在多个选项中做选择时，Peaks 可以并行生成多个候选方案，让用户比较真实效果。

示例：

- 产品：多个 PRD 或工作流方向；
- 设计：多个高保真 HTML 原型；
- 研发：多个技术 POC；
- QA：多种测试策略或矩阵；
- 重构：多种现代化策略。

该模式适用于：

- 选项之间差异显著；
- 用户无法仅凭文本评估；
- 真实效果很重要；
- 借助 AI/群体执行的重启成本较低。

该模式不适用于：

- 高风险热修；
- 不可逆数据库迁移；
- 安全敏感的生产变更；
- 没有回滚计划的大型重构。

## 12. 项目重构工作流

重构应该是一等场景。

技能不应该立即编辑代码。它应该先：

1. 阅读当前代码和已知产物；
2. 识别业务边界和痛点；
3. 识别技术约束；
4. 产出多个重构选项；
5. 标注成熟度、风险、成本和回滚路径；
6. 让人选择；
7. 创建重构执行计划；
8. 在实现前生成 QA 回归保护。

重构建议应该优先考虑当前业务和团队的适配性，而不是抽象的技术潮流。

## 13. 技能沉淀

角色技能交付中间产物后，应该识别可复用操作，并询问是否将其持久化为技能。

示例：

- 会员系统 PRD 生成器；
- RBAC 测试矩阵生成器；
- 状态机技术设计模式；
- B 端表格页原型模式；
- 生产状态迁移事故清单。

如果用户批准：

1. 创建技能候选产物；
2. 运行技能创建；
3. 运行 Darwin 式技能优化；
4. 生成测试提示词；
5. 本地安装；
6. 可选地与团队成员共享。

这会形成一个持续的组织学习闭环。

## 14. OpenSpec 关系

当前方向已经超出 OpenSpec 本身。OpenSpec 可以作为其中一种规格管理和变更拆解工具，但不应该定义 Peaks AI 协作技能体系的边界。

### 14.1 适合交给 OpenSpec 管理的内容

OpenSpec 适用于：

- 向现有项目添加新的技能家族能力；
- 改变某个具体技能或编排入口的行为；
- 定义新的产物类型；
- 引入角色技能边界；
- 添加同步、上下文、问题和体验产物的功能要求；
- 跟踪实现任务。

未来的 OpenSpec 变更可以包含：

```text
openspec/changes/peaks-ai-collaboration-skill-system/
├── proposal.md
├── design.md
├── tasks.md
└── specs/
    ├── peaks-sdd.md
    ├── peaks-artifact.md
    ├── peaks-context.md
    ├── peaks-sync.md
    ├── peaks-product.md
    ├── peaks-design.md
    ├── peaks-rd.md
    ├── peaks-qa.md
    └── peaks-issue.md
```

### 14.2 不应该由 OpenSpec 承载的内容

整体愿景、产品哲学、技能产品线边界、独立产物 Git 仓策略、团队协作方法论和长期组织学习闭环，不应该被压缩成一个 OpenSpec 变更。OpenSpec 更适合作为某个阶段、某个技能、某个协议能力的落地规格。

如果用 OpenSpec 落地，可以按阶段拆解为：

1. 协议内核：artifact + context + trace schemas；
2. 角色技能边界：product/design/RD/QA 拆分；
3. 独立产物 Git 仓与业务代码仓同步连接器；
4. 体验产物与高保真原型工作流；
5. 向 RD/QA 传播变更；
6. 问题产物与 QA 到 RD 修复闭环；
7. 事故/复盘工作流；
8. 技能沉淀闭环；
9. 编排器模式与场景路由。

### 14.3 必需的 OpenSpec 检查点

对于每个分阶段变更：

- proposal 必须明确目标、范围和价值；
- specs 必须定义可测试行为和产物契约；
- design 必须描述架构、风险、依赖和迁移；
- tasks 必须可执行且可审查；
- archive 必须在实现后更新当前行为规格。

## 15. 现有 peaks-sdd 与 harness 的迁移关系

当前方向不再局限于现有 peaks-sdd harness，但可以参考其中已经验证过的门禁、产物和执行经验。新的技术方案应该生成一组全新的 Peaks 协作技能；现有 `peaks-sdd` 的部分实现可以作为参考，而不是作为必须延续的架构边界。

### 15.1 可参考的现有 harness 要求

当前 peaks-sdd 门禁要求：

- 交互式脑暴记录；
- PRD；
- PRD 确认；
- OpenAPI/no-API 产物；
- 设计规格；
- 视觉设计产物；
- 设计确认；
- 系统设计；
- 系统设计确认；
- 测试计划；
- 群体任务图；
- waves；
- briefs；
- handoffs；
- 执行代理；
- 模块报告；
- 代码审查；
- 安全审查；
- 单元测试报告；
- 覆盖率摘要；
- 功能/业务/性能/运行时冒烟报告；
- QA 轮次；
- 验收报告；
- 最终报告。

拟议设计保留这些理念，但将它们扩展为更广泛的协议。

### 15.2 新体系需要的门禁能力

新体系不应该简单继承旧 harness 的固定文件检查，而应该把门禁能力抽象为类型化产物和契约检查。

新的门禁最终应该验证：

- 人类/模型同步状态；
- Schema 有效性；
- 独立产物 Git 仓与业务代码仓同步状态；
- 上下文胶囊新鲜度；
- 追踪图完整性；
- 产品/设计/研发/QA/问题产物链接；
- 变更影响传播；
- QA 资产更新证据；
- 问题复现证据；
- 人类确认记录；
- 技能沉淀提案。

### 15.3 中间产物迁移规则

新的技能组可以舍弃旧的 `.peaks/changes/<change-id>/` 目录作为业务代码仓内的固定布局，但不应该舍弃其中沉淀出来的中间产物类型。这些产物应该迁移到独立产物 Git 仓，并转换为协议化 artifact、trace、context 和 evidence。

迁移映射：

| 当前 harness 产物 | 未来协议产物 |
| --- | --- |
| `product/prd.md` | `artifact.prd/human.md` |
| `product/prd-confirmation.md` | manifest + human.md 中的确认记录 |
| `design/design-spec.md` | `artifact.design-spec/human.md` |
| `design/approved-preview.html` | `experience/prototype.html` |
| `architecture/system-design.md` | `artifact.tech-design/human.md` |
| `qa/test-plan.md` | `artifact.test-plan/human.md` |
| `swarm/task-graph.json` | 追踪图 + 群体执行产物 |
| `swarm/briefs/*.md` | 角色执行 brief |
| `swarm/handoffs/*.md` | 追踪图中的交接记录 |
| `review/code-review.md` | 审查产物 |
| `security/security-report.md` | 安全产物 |
| `qa/*report.md` | QA 证据产物 |
| `final-report.md` | 最终交付摘要 + 匹配报告 |

## 16. MVP 建议

本设计有意保持较宽范围。MVP 应该避免一次性构建所有内容。

推荐的 MVP 顺序：

1. Peaks 协作协议内核：
   - 产物文件夹布局；
   - `human.md` + `model.json`；
   - manifest；
   - Schema 校验；
   - 追踪链接。
2. 上下文胶囊：
   - 产品胶囊；
   - RD/QA 切片；
   - 过期元数据。
3. 角色拆分：
   - `peaks-product`；
   - 轻量版 `peaks-design`；
   - `peaks-rd`；
   - `peaks-qa`。
4. 体验产物：
   - 高保真 HTML 原型；
   - 布局和交互模型；
   - QA 场景种子。
5. 变更传播：
   - 产品变更 → RD/QA 影响包。
6. 问题产物：
   - QA 问题 → RD 复现/修复 → QA 回归。
7. 独立产物 Git 仓同步：
   - 产物仓与业务代码仓指针；
   - 拉取/推送快照；
   - 提案/差异流程。
8. 技能沉淀：
   - 检测候选项；
   - 询问用户；
   - 生成技能候选。
9. 事故/复盘：
   - 复用问题协议；
   - 增加缓解和复盘产物。

## 17. 已确认决策与未决问题

### 17.1 已确认决策

1. `peaks-artifact`、`peaks-context` 和 `peaks-sync` 两者兼具：既可以作为用户可调用技能，也可以作为其他角色技能复用的内部能力。
2. 不再以外部协作文档作为默认事实源；中间产物、协作记录和确认记录优先进入独立产物 Git 仓。
3. 人类/模型产物应该实时同步；引入 Git 后，未提交或未合并的变更不会影响已确认事实源。
4. 设计技能命名为 `peaks-design`，因为它不只是输出高保真 UI，还要覆盖用户体验、交互、视觉方向和体验评估。
5. 旧的 `.peaks/changes/<change-id>/` 固定布局暂时不作为新体系的兼容表面，但其中有价值的中间产物类型应迁移到独立产物 Git 仓。
6. 新方案要生成一组全新的 Peaks 协作技能；当前 `peaks-sdd` 的实现可以参考，但不应限制新架构。
7. OpenSpec 和 harness 是可借鉴的落地工具与历史经验，不再是该设计的概念边界。

### 17.2 仍需澄清的问题

1. 第一个要稳定的产物 Schema 应该从哪个落地场景切入：PRD、experience、issue，还是 change-impact？这个问题本质是在问 MVP 先固化哪一种“产物契约”，而不是只选一个文档名称。
2. 独立产物 Git 仓应该如何组织：每个业务项目一个产物仓，还是多个项目共享一个组织级产物仓？
3. 产物仓的分支、提交、审查和确认模型应该如何设计，才能同时适配单人模式和团队模式？
4. 旧 `.peaks/changes/<change-id>/` 中哪些产物类型最值得优先迁移到新协议？

## 18. 本设计的验收标准

如果本设计满足以下条件，则可接受：

- 保留用户的低学习成本；
- 同时支持单人自动化和团队协作；
- 定义独立和编排两种技能模式；
- 使用厂商中立产物；
- 区分面向人阅读和面向模型读取的输出；
- 支持精确上下文传递，而不是倾倒完整对话；
- 支持独立产物 Git 仓与业务代码仓事实源分离；
- 将 `peaks-design` 作为覆盖用户体验、交互和视觉的一等角色；
- 将产品变更传播到研发和 QA；
- 支持 QA 问题产物和生产事故复用；
- 保持人类对最终决策负责；
- 抬高质量下限，而不是承诺所有人都获得相同的专家级结果；
- 可以在需要时拆解为 OpenSpec 兼容的分阶段变更；
- 可以吸收当前 harness SDD 门禁经验，并迁移为新体系的协议化门禁。

## 19. 下一轮脑暴方向

这一轮已经把体系边界、事实源、角色命名、迁移方向和 OpenSpec/harness 关系重新定了下来。下一轮更值得继续展开的，不是再讨论“要不要做”，而是讨论“怎么让它稳定地跑起来”。

### 19.1 先继续展开的三条主线

#### 1）协议内核

最值得先稳定的是协议内核，而不是单个技能的实现细节。协议内核至少要回答：

- 一个产物如何被唯一标识；
- 一个产物如何被版本化；
- 一个产物如何和上下游产物建立 trace；
- 一个产物如何表达 human/model 双表示；
- 一个产物如何表达证据、确认和审批。

这条主线里最核心的是 `manifest`、`trace`、`context`、`approval`、`evidence`。

#### 2）协作运行时

协议有了以后，还需要一个协作运行时来定义如何工作：

- 分支怎么开；
- 提交怎么写；
- 审查怎么做；
- 确认怎么落；
- 冲突怎么处理；
- 回滚怎么执行；
- 谁拥有最终确认权。

这条主线决定了独立产物 Git 仓是不是能真正替代“只靠对话同步”的协作方式。

#### 3）技能生命周期

新体系不是一批静态技能，而是一组会持续演化的技能家族。还应该补清楚：

- 技能怎么创建；
- 技能怎么版本升级；
- 技能怎么被标记弃用；
- 技能怎么迁移到新协议；
- 技能怎么共享给别的项目；
- 技能怎么被回收或降级。

### 19.2 最值得优先固化的 5 个核心契约

如果要选下一批最该稳定的产物/流程，我会优先考虑这 5 个：

1. `manifest`
   - 解决“这是哪个产物、属于谁、当前版本是什么、与谁相关”的问题。
2. `change-impact`
   - 解决“一个变更会影响哪些研发和 QA 资产”的问题。
3. `approval-record`
   - 解决“谁在什么条件下确认了什么”的问题。
4. `context-capsule`
   - 解决“下游角色到底需要哪些最小背景”的问题。
5. `issue` / `incident`
   - 解决“问题如何进入复现、修复、回归、复盘链路”的问题。

如果只能先固化一个，我会建议先把 `change-impact` 的契约想清楚。原因是它最容易把产品变更、研发动作和 QA 验证串成一个最小闭环；`PRD` 更像输入，`experience` 更像表达，`issue` 更像反馈，而 `change-impact` 更接近让体系真正动起来的连接件。

### 19.3 还缺的章节骨架

如果继续扩写设计稿，下一批值得补的章节可以是：

- **身份与权限模型**：谁可以创建、修改、确认、发布、回滚、归档。
- **版本与分支模型**：产物仓如何表达主线、分支、候选、草稿、正式版。
- **冲突与合并策略**：人类修改和模型修改冲突时如何处理。
- **确认与审批策略**：哪些变更必须 human confirmed，哪些可以自动通过。
- **可观测性与审计**：如何追踪每一步是谁做的、为什么做、证据在哪。
- **弃用与迁移策略**：旧技能、旧产物、旧布局如何逐步退场。
- **模板与脚手架**：不同角色技能初始骨架如何快速生成。

### 19.4 可以直接继续追问的设计问题

- 产物仓应该按项目建，还是按组织建？
- 一个产物可以同时属于多个技能吗？
- `change-impact` 应该由 product 生成，还是由 rd/qa 联合生成？
- `approval-record` 是附在 manifest 里，还是作为独立 artifact？
- skills 的发布单位应该是一个目录、一个仓，还是一个可安装包？
- 什么时候应该强制 human confirmation，什么时候可以只保留 model verification？
- 旧的 `peaks-sdd` 应该作为兼容入口保留多久？

### 19.5 如果下一轮要写成正式章节

比较适合优先落成正式章节的，不是更多抽象口号，而是下面这四个：

1. **协议内核设计**：manifest、trace、context、approval、evidence。
2. **产物仓运行模型**：分支、提交、审查、确认、冲突、回滚。
3. **技能生命周期设计**：创建、版本、弃用、迁移、共享。
4. **变更影响与问题闭环**：change-impact、issue、incident、regression。
